## Overview

The **web app** (`apps/web`) is the primary development target and reference implementation. The **iOS app** (`apps/ios`) is a native SwiftUI implementation that mirrors the web for canopy and manual wingsuit modes, but **does not yet support wingsuit auto mode**.

## Divergence Risks

### 1. Auto Solver Staleness (High Risk)

The iOS `PatternEngine.swift` contains a `solveWingsuitAuto()` implementation (~542 lines from line 800 onward) that uses an **older architecture**:
- **Reverse-solve approach** instead of the web's forward-parametric sweep
- **No jump-run slot system** â€” no slick group awareness, no distance placement
- **Simpler scoring** â€” no rectangular shape preference, no radius annulus
- **Fewer diagnostics** â€” the `WingsuitAutoDiagnostics` struct in iOS is missing fields like `headingSource`, `placementMode`, `headwindComponentKt`, `canopyReturnMarginFt`, `shapePenalty`

**Mitigation:** The iOS code correctly gates auto mode off via `iosWingsuitAutoModeEnabled = false`. When it is time to port, the entire auto solver should be rewritten from the web TypeScript as the reference, not incrementally patched.

### 2. Contract Drift (Medium Risk)

The `WingsuitAutoOutput` structs differ between platforms:
- **Web** (`@landing/ui-types`): includes `resolvedJumpRun: ResolvedJumpRun | null`, `landingNoDeployZonePolygon`, `downwindDeployForbiddenZonePolygon`, full `diagnostics` with ~16 fields
- **iOS** (`Contracts.swift`): omits `resolvedJumpRun` (uses `forbiddenZonePolygon` directly), diagnostics has only 8 fields, lacks `landingNoDeployZonePolygon` and `downwindDeployForbiddenZonePolygon`

The parity fixture tests help catch basic solver output differences but **do not cover contract shape changes**. When porting auto mode, the iOS contracts need to be updated to match the web output shape.

### 3. Fixture Sync (Low Risk)

The `npm run ios:fixtures:sync` pipeline copies generated fixture JSON into the Swift test fixtures directory. If an engineer modifies the TS engine and forgets to run `ios:fixtures:sync`, the Swift parity tests will pass against stale data. The pipeline script (`ios_pipeline.sh`) mitigates this by running sync as part of the full build.

## Recommended Porting Order

When enabling wingsuit auto mode on iOS:

1. **Update `Contracts.swift`** â€” add missing output fields to match web's `WingsuitAutoOutput`, `ResolvedJumpRun`, and `WingsuitAutoDiagnostics`
2. **Port `solveWingsuitAuto()`** â€” translate the web's forward-parametric solver from `packages/engine/src/index.ts` (lines 1966â€“2274) to Swift, including jump-run resolution, forward route generation, hard filters, and scoring
3. **Update fixture generation** â€” add new auto mode fixtures covering normal and distance placement
4. **Flip the gate** â€” set `iosWingsuitAutoModeEnabled = true` in `LandingStore.swift`
5. **Map rendering** â€” implement auto-mode zone overlays in `MapKitLandingMapView.swift` using the existing `LandingMapViewProtocol` (which already declares `autoOutput: WingsuitAutoOutput?`)
6. **Test** â€” run `npm run ios:pipeline` and manual verification
