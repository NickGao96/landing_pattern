# Web / iOS Feature Status

> Detailed feature parity comparison between the web and iOS apps.
> Last audited against code as of April 2026.

## Overview

The **web app** (`apps/web`) is the primary development target and reference implementation. The **iOS app** (`apps/ios`) is a native SwiftUI implementation that mirrors the web for canopy and manual wingsuit modes and now includes a Swift port of wingsuit auto mode.

Both apps have **independent engine implementations** — TypeScript (`packages/engine`) and Swift (`LandingPatternCore`). Parity is verified through JSON fixture tests generated by the TypeScript engine and consumed by Swift unit tests.

## Divergence Risks

### 1. Auto Solver Drift (Medium Risk)

The Swift wingsuit auto solver is a port of the TypeScript forward-parametric solver, not shared code. Keep the TypeScript implementation as the source of truth and update `packages/fixtures/engine/wingsuit-auto-fixtures.json` whenever the web solver behavior changes. `EngineParityTests.testWingsuitAutoParityAgainstTypeScriptFixtures` should pass before changing iOS auto behavior.

### 2. UI Parity Drift (Medium Risk)

The iOS engine/input contract supports normal and distance placement, manual/auto heading, reciprocal constraints, and jump-run assumptions. The iOS SwiftUI controls are still more compact than the web controls, so UI parity should be audited before declaring full product parity.

### 3. Fixture Sync (Low Risk)

The `npm run ios:fixtures:sync` pipeline copies generated fixture JSON into the Swift test fixtures directory. If an engineer modifies the TS engine and forgets to run `ios:fixtures:sync`, the Swift parity tests will pass against stale data. The pipeline script (`ios_pipeline.sh`) mitigates this by running sync as part of the full build.

## Recommended Follow-Up

1. Expand iOS SwiftUI controls for the full web jump-run settings surface.
2. Add distance-placement auto fixtures to `generate-auto-fixtures.mjs`.
3. Keep running `npm run ios:core:test` after fixture changes and `npm run ios:pipeline` when local Xcode is healthy.
